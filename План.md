### Этап 0. Подготовка и Telegram-инфраструктура

1. [ ] Создать бота через `@BotFather`, получить `BOT_TOKEN`.
2. [ ] Зарезервировать имя Mini App (`/newapp` в BotFather) — понадобится HTTPS-ссылка, но её можно добавить позже.
3. [ ] Создать тестовый чат/канал для быстрой проверки открытия WebApp на реальном устройстве (эмуляторы часто врут насчёт поведения WebView).

### Этап 1. Подключение Telegram SDK на фронте

1. Установить `@twa-dev/sdk` (или подключить `https://telegram.org/js/telegram-web-app.js` напрямую в `index.html`).
2. В точке входа (`src/main.tsx`) вызвать `WebApp.ready()` и `WebApp.expand()`.
3. Получить `initData` и `initDataUnsafe` — пока просто вывести в консоль, чтобы убедиться, что данные приходят (это будет видно только при открытии внутри реального Telegram, не в браузере).
4. Прокинуть `themeParams` Telegram в CSS-переменные Tailwind (цвет фона, текста, кнопок) — чтобы приложение подстраивалось под тему пользователя (светлая/тёмная, разные клиенты).
5. Заменить кастомные кнопки "Назад" (`ArrowLeft` + `onClick`) на нативный `WebApp.BackButton` — показывать/скрывать в зависимости от `screen`, вешать `onClick` на `setScreen(...)`.
6. Опционально: `MainButton` для действий типа "Сохранить тренировку" вместо своей кнопки внизу экрана.

**Критерий готовности этапа:** приложение открывается внутри Telegram (через тестовую ссылку на `ngrok`/Vercel preview), разворачивается на весь экран, кнопка "Назад" в Telegram работает.

### Этап 2. Валидация пользователя (замена AuthScreen)

1. Удалить/не использовать `AuthScreen.tsx` в текущем виде — email/password логин не нужен.
2. На бэкенде (в уже существующем `supabase/functions/server/index.tsx`) написать функцию проверки `initData`:
    - разобрать query-string из `initData`
    - вычислить HMAC-SHA256 подпись секретным ключом, производным от `BOT_TOKEN`
    - сравнить с полем `hash` из `initData`
    - это стандартный алгоритм, описанный в документации Telegram — займёт один эндпоинт `POST /auth/telegram`
3. При успешной валидации — доставать `telegram_user_id` из `initDataUnsafe.user.id` и либо создавать запись пользователя в БД, либо выдавать свой JWT/сессионный токен для дальнейших запросов.
4. На фронте: на старте приложения отправлять `initData` на этот эндпоинт, сохранять полученный токен в памяти (не в localStorage, если токен короткоживущий — но для простого MVP можно и в localStorage).

**Критерий готовности:** можно достоверно сказать "этот запрос пришёл от пользователя Telegram с ID X", без пароля/регистрации.

### Этап 3. Перенос данных с localStorage на бэкенд

1. Спроектировать структуру данных под `kv_store` (или завести нормальные таблицы в Postgres через Supabase, если данные структурированные — тренировки/упражнения это оправдывает):
    - `workouts:{telegram_user_id}` → список тренировок (или отдельная таблица `workouts` с полем `user_id`)
2. Написать CRUD-эндпоинты в той же Hono-функции:
    - `GET /workouts` — список тренировок пользователя
    - `POST /workouts` — создать
    - `PUT /workouts/:id` — обновить (в т.ч. добавление подходов/упражнений)
    - `DELETE /workouts/:id` — удалить
3. В `App.tsx` заменить:
    - `localStorage.getItem('gym-workouts')` → `fetch('/workouts', {headers: {Authorization: ...}})`
    - `saveWorkouts()` → `PUT/POST` запрос вместо `localStorage.setItem`
4. Добавить состояния загрузки/ошибок (`isLoading`, `error`) — сейчас их нет вообще, т.к. localStorage синхронный.
5. Решить, что делать offline: либо оставить localStorage как временный кэш с последующей синхронизацией, либо просто требовать сеть (для MVP это ок).

**Критерий готовности:** удаление приложения из Telegram и повторная установка → данные тренировок на месте (значит, хранятся не локально).

### Этап 4. VoiceRecorder — проверить и адаптировать

1. Посмотреть `VoiceRecorder.tsx` — на чём сейчас основано распознавание речи (Web Speech API браузера или сторонний сервис).
2. Web Speech API **не работает надёжно во всех Telegram WebView** (особенно на iOS) — если он используется, придётся либо:
    - записывать аудио и отправлять на бэкенд для распознавания (Whisper API и т.п.), либо
    - сделать честный fallback на ручной ввод, если API недоступен.

### Этап 5. Очистка мёртвого веса

1. Удалить неиспользуемые зависимости: `@mui/material`, `@emotion/*`, `@popperjs/core`, `react-popper`, `react-router`, `react-slick`, `react-responsive-masonry` — то, что реально не импортируется нигде в `src/`.
2. Прогнать `vite build` с анализом бандла (`rollup-plugin-visualizer` или просто `vite build --report`), убедиться, что итоговый размер разумный — для WebView на слабых Android-устройствах это важно.

### Этап 6. Деплой и подключение к боту

1. Задеплоить `vite build` на статический хостинг с HTTPS (Vercel/Cloudflare Pages/Netlify — любой подходит).
2. Задеплоить Supabase Edge Function (`supabase functions deploy`).
3. Указать ссылку на фронтенд в BotFather как Web App URL для кнопки меню бота или инлайн-кнопки `web_app`.
4. Прогнать полный сценарий вживую: открытие через бота → авторизация по initData → создание тренировки → перезапуск → данные на месте.

### Этап 7. Полировка под мобильный WebView

1. Проверить safe area insets (вырезы/чёлки) — использовать `viewportStableHeight` из SDK вместо `100dvh`, где уже используется (`renderNewWorkout` сейчас на `100dvh` — стоит перепроверить на реальных устройствах).
2. Отключить pull-to-refresh браузера, если мешает (частая проблема в TMA).
3. Проверить haptic feedback (`WebApp.HapticFeedback`) на ключевых действиях (завершение тренировки, удаление) — это то, что отличает "просто сайт" от "приложения, которое ощущается нативным".
