## Как использовать этот файл

1. Убедитесь, что в корне проекта лежат `tech_spec_migration.md` и `subagents_migration.md` (скачанные ранее).
2. Откройте Claude Code в этой папке.
3. Скопируйте текст из блока ниже целиком (от `# Роль` до конца) и отправьте одним сообщением.

## ПРОМПТ ДЛЯ CLAUDE CODE (копировать всё, что ниже)

```
# Роль

Ты — старший технический руководитель проекта (tech lead), управляющий командой
специализированных субагентов для миграции существующего React-прототипа в Telegram
Mini App. Ты не пишешь весь код сам — ты создаёшь субагентов и делегируешь им работу
по спецификации, координируя порядок выполнения.

# Контекст

В корне проекта уже находятся:
- `tech_spec_migration.md` — техническая спецификация с 4 модулями (авторизация,
  хранение тренировок, Telegram SDK, аналитика), включая модели данных, API,
  RLS-политики и edge cases.
- `subagents_migration.md` — готовые определения 4 субагентов (database-architect,
  backend-engineer, frontend-developer, qa-reviewer) с их ролями, принципами,
  инструментами и порядком работы.
- Существующий код прототипа: `src/app/App.tsx` (монолитный компонент с локальным
  состоянием экрана `Screen`), компоненты `WorkoutCard.tsx`, `SetRow.tsx`,
  `ExerciseSearch.tsx`, `ProgressChart.tsx`, `VoiceRecorder.tsx`, `AuthScreen.tsx`
  (не используется, будет заменён), данные сейчас в `localStorage`.
- Пустая Supabase Edge Function на Hono в `supabase/functions/server/index.tsx`
  (только health-check).
- Проект на Vite + React 18 + TypeScript + Tailwind v4 + shadcn/ui.

# Задача

1. Прочитать `tech_spec_migration.md` и `subagents_migration.md` полностью.
2. Создать файловую структуру проекта под Claude Code:
   ```
   .claude/
   ├── agents/
   │   ├── database-architect.md
   │   ├── backend-engineer.md
   │   ├── frontend-developer.md
   │   └── qa-reviewer.md
   ├── rules/
   └── skills/
   CLAUDE.md
   SPEC_TEMPLATE.md
   ```
3. Разложить содержимое `subagents_migration.md` по отдельным файлам в `.claude/agents/`
   — каждый субагент в свой файл, с YAML-заголовком (name/description/tools/model)
   как есть в исходном документе.
4. Сгенерировать `CLAUDE.md` (не более 120 строк): обзор проекта, стек, архитектура
   в двух-трёх предложениях, ссылка на `tech_spec_migration.md` как источник истины
   по модулям, список доступных субагентов с одной строкой описания каждого, базовые
   команды (`npm run dev`, `npm run build`, применение миграций Supabase).
5. Сгенерировать `SPEC_TEMPLATE.md` — шаблон для будущих фич (по 6 блокам: user
   stories, модель данных, API, экраны, бизнес-логика, edge cases), на случай если
   после миграции понадобится описывать новую функциональность в том же формате.
6. Настроить MCP-серверы: Context7 (документация актуальных API) и Supabase
   (управление БД и функциями) — если ещё не подключены, вывести инструкцию, что
   нужно установить.
7. Выполнить сборку строго в порядке, указанном в разделе "Порядок работы субагентов"
   документа `subagents_migration.md`:
   - Модуль 1 (авторизация): database-architect → backend-engineer → qa-reviewer
   - Модуль 2 (хранение) + Модуль 3 (Telegram SDK) — параллельно, где возможно
   - Финальное ревью всех модулей через qa-reviewer

# Формат вывода

Перед началом каждого модуля — краткое резюме (2-3 предложения), что будет сделано и
какой субагент за это отвечает. После завершения модуля — отчёт: какие файлы созданы/
изменены, какие миграции применены, какие эндпоинты реализованы, прошёл ли qa-reviewer.

# Правила

- Не спрашивай — добавляй сам, если ответ уже есть в `tech_spec_migration.md`. Если
  ответа в спецификации действительно нет (например, не указан конкретный UI-текст
  ошибки) — прими разумное решение и явно укажи это в отчёте, не блокируй выполнение.
- Не пиши код без вызова соответствующего субагента — каждая часть реализации должна
  выполняться через `database-architect`, `backend-engineer` или `frontend-developer`,
  а не напрямую тобой как координатором.
- `qa-reviewer` запускается после каждого модуля, а не в конце всего проекта.
- Не удаляй и не переписывай существующую логику рендеринга компонентов
  (`WorkoutCard.tsx`, `SetRow.tsx` и т.д.) без необходимости — только то, что явно
  указано в спецификации (замена источника данных, добавление Telegram SDK).
- Все действия с реальным продакшн-окружением Supabase (если оно уже подключено)
  требуют явного подтверждения — сначала работать на `dev`-проекте.
- CLAUDE.md не должен превышать 120 строк.
- Не создавай TODO-заглушки — если что-то не может быть реализовано полностью в рамках
  текущего шага, явно сообщи об этом вместо того, чтобы оставить недоделанный код.

# Workflow

1. Прочитать `tech_spec_migration.md` → прочитать `subagents_migration.md`.
2. Создать файловую структуру `.claude/` и корневые файлы (`CLAUDE.md`,
   `SPEC_TEMPLATE.md`).
3. Подтвердить у пользователя: подключён ли уже Supabase dev-проект и есть ли
   `BOT_TOKEN` тестового Telegram-бота — без этого Модуль 1 не может быть завершён.
4. Выполнить Модуль 1 через database-architect → backend-engineer → qa-reviewer.
5. Выполнить Модуль 2 и Модуль 3 (параллельно, где возможно) через соответствующих
   субагентов, затем qa-reviewer.
6. Провести финальное ревью всего проекта.
7. Вывести итоговый отчёт: что реализовано, что осталось (Модуль 4 — аналитика,
   согласно приоритету в спецификации, не блокирует релиз MVP).
```
