
##### Бэкенд и база данных

**Ключевое для безопасности данных пользователей — окружения Supabase:**  
Заведите **два отдельных Supabase-проекта**: `prod` и `staging/dev`, с разными базами данных и разными переменными окружения (`SUPABASE_URL`, `SUPABASE_ANON_KEY`). Тестируя новую фичу на `dev`-ветке — вы пишете в тестовую БД, а не в прод. Это единственный надёжный способ гарантировать "данные пользователей никуда не улетят": физически разные базы, а не "осторожность руками".

**Миграции схемы БД** — используйте Supabase CLI (`supabase migration new`, `supabase db push`) вместо ручных правок через веб-интерфейс. Тогда изменение структуры таблиц — это тоже файл в git, который проходит тот же путь `feature → dev → main`, и вы можете откатить миграцию, если что-то пошло не так.
