- **Стабильный анонимный/пользовательский ID с самого начала** (у вас уже будет `telegram_user_id` из этапа авторизации) — вся А/Б-инфраструктура строится на детерминированном распределении по ID (например, хэш от ID % 100 → группа).
- **Feature flags как отдельный слой**, не хардкод. Даже простейший вариант: таблица `feature_flags` в Supabase или JSON-конфиг, который фронт запрашивает при старте — вместо `if (Math.random() > 0.5)` в компоненте. Готовые сервисы (PostHog features flags, GrowthBook) тоже подходят и часто идут в одной связке с аналитикой из пункта 3.
- **Событийная аналитика (пункт 3) — это предпосылка A/B**, без неё тестировать нечего: сначала нужно фиксировать события, потом сравнивать метрики между группами.
